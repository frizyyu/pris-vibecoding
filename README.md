# PRIS MVP

PRIS MVP — прототип веб-приложения для управления задачами и контроля сроков выполнения, созданный для лабораторной работы по теме «Разработка и анализ прототипа веб-приложения для управления задачами и контроля сроков выполнения с использованием средств вайб-кодинга».

## Цель проекта

Проект нужен как наглядный MVP, который можно показать на защите и использовать для анализа:

- как быстро собрать рабочий fullstack-прототип на современном стеке;
- как организовать базовый цикл работы с задачами и дедлайнами;
- как проверить жизнеспособность идеи без усложнения архитектуры.

## Какие боли пользователя закрывает проект

Приложение помогает в сценарии, когда задачи ведутся разрозненно и сроки начинают теряться.

Что оно закрывает:

- нет единого списка задач и трудно увидеть, что уже в работе;
- дедлайны забываются, а просрочка замечается слишком поздно;
- сложно быстро понять, что сейчас приоритетнее;
- нет удобного способа отфильтровать активные, выполненные или просроченные задачи;
- трудно быстро найти нужную задачу по названию или описанию;
- нет быстрой демонстрации статуса проекта для учебной защиты или внутреннего показа.

## Технологии

- Next.js App Router
- TypeScript
- Tailwind CSS
- Prisma
- SQLite

## Возможности MVP

- список задач на главной странице;
- создание, редактирование, удаление и завершение задач;
- хранение `title`, `description`, `deadline`, `priority`, `status`, `createdAt`, `updatedAt`;
- поиск по `title` и `description`;
- совместная работа поиска, фильтрации и сортировки;
- выделение просроченных задач;
- статистика по задачам;
- seed-данные для демонстрации.

## Локальный запуск

```bash
npm install
npm run db:setup

```

```text
http://localhost:3000
```

- `npm run db:setup` — полный запуск базы для локальной разработки

## Архитектура проекта

Проект собран как одно Next.js-приложение с App Router: UI, route handlers и серверная логика находятся в одном репозитории. Архитектура намеренно простая: страницы читают данные через `taskService`, а доступ к базе изолирован в Prisma repository.

Поток данных такой:

1. страницы и route handlers получают входные данные;
2. `task-query` разбирает query params и form data;
3. `taskService` применяет доменные правила и валидацию;
4. Prisma repository читает и записывает данные в SQLite.

## Основные модули

- `app/` — страницы, route handlers и серверные entry points
- `components/` — UI-компоненты, карточки и формы
- `src/lib/tasks.ts` — доменная модель, фильтрация, сортировка, overdue-логика и валидация
- `src/lib/task-service.ts` — единая точка создания сервиса задач
- `src/lib/task-query.ts` — разбор query params и form data
- `src/lib/task-metadata.ts` — общие справочники статусов, приоритетов, фильтров и подписей
- `src/lib/task-presentation.ts` — presentation helpers для статистики, дат и текстовых форм
- `src/lib/task-repository.prisma.ts` — работа с Prisma и SQLite
- `prisma/` — схема, миграции и seed

## Где находится ключевая логика

- `src/lib/tasks.ts` — валидация, нормализация, вычисление `overdue`, фильтрация и сортировка
- `src/lib/task-query.ts` — преобразование HTTP-параметров и form data в понятные приложению структуры
- `src/lib/task-repository.prisma.ts` — подготовка данных для базы и Prisma CRUD
- `components/tasks/task-board.tsx` — главная панель управления задачами
- `components/tasks/task-card.tsx` и `components/tasks/task-form.tsx` — базовый UI задач

## Как захостить проект

### Важное ограничение перед деплоем

Проект использует SQLite. Это значит, что для полноценной работы нужен хостинг с постоянной файловой системой или выделенным диском/volume. Для Vercel такой вариант не подходит как основной production-хостинг, потому что локальная файловая система там эфемерная.

### Рекомендуемый вариант для хостинга

Самый простой и надёжный способ — VPS или облачный сервер с Linux, Node.js и постоянным диском.

### Какие ресурсы нужны

Для MVP достаточно:

- 1 vCPU;
- 1 GB RAM;
- 5-10 GB SSD;
- Ubuntu 22.04+;
- Node.js 20+;
- доступ по SSH;
- домен и reverse proxy по желанию.

### Пошаговая инструкция по хостингу на VPS

1. Подготовить сервер и установить Node.js 20+, npm и Git.
2. Клонировать репозиторий:

```bash
git clone https://github.com/frizyyu/pris-vibecoding.git
cd pris-vibecoding
```

3. Установить зависимости:

```bash
npm install
```

4. Создать `.env` на основе `.env.example` и убедиться, что путь к SQLite корректен.
5. Подготовить базу и seed-данные:

```bash
npm run db:setup
```

6. Собрать проект:

```bash
npm run build
```

7. Запустить production-сервер:

```bash
npm run start
```

8. Для фонового запуска использовать `pm2`:

```bash
npm install -g pm2
pm2 start npm --name pris -- run start
pm2 save
pm2 startup
```

9. При необходимости настроить Nginx как reverse proxy на `localhost:3000`.
10. Для HTTPS подключить Let's Encrypt через Certbot.

### Минимальная схема публикации

- приложение работает на `localhost:3000`;
- Nginx принимает внешний трафик по `80/443`;
- SQLite хранится на диске сервера;
- PM2 держит процесс Next.js запущенным после перезагрузки.

## Ограничения текущего решения

- это MVP для лабораторной работы, а не production-система;
- отсутствуют авторизация, роли и многопользовательская работа;
- используется локальная SQLite база;
- `overdue` вычисляется динамически по дедлайну и статусу, а не хранится отдельным полем;
- нет фоновых задач, уведомлений и истории изменений;
- структура упрощена ради наглядности анализа, а не ради масштабирования.
