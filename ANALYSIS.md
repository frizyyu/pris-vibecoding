# ANALYSIS

## Что тестировалось

Проверены основные пользовательские сценарии на локальном dev-сервере после заполнения базы seed-данными:

- загрузка главной страницы;
- создание задачи;
- редактирование задачи;
- удаление задачи;
- отметка выполнения через смену статуса на `DONE`;
- корректность `completedAt` при завершении задачи;
- работа дедлайнов и `overdue`;
- поиск по `title` и `description`;
- фильтрация по статусу, приоритету и просроченности;
- сортировка по приоритету и дедлайну;
- валидация невалидного ввода;
- повторный запуск `build` и `dev` после внесённых исправлений.

## Какие проблемы были найдены

1. Ошибки валидации в API возвращались как `500 Internal Server Error`, хотя по смыслу это пользовательские ошибки ввода.
2. Ошибки валидации в form routes не давали понятной обратной связи и могли уводить пользователя в error boundary вместо возврата к форме.
3. POST-роуты использовали обычный redirect без явного `303 See Other`, что хуже для UX после form submit.
4. Во время QA проявился нестабильный dev-state при смешении старого `dev`-процесса и новой сборки: `.next` мог оказаться в битом состоянии с отсутствующим чанком.
5. В проекте нет автоматизированных e2e-тестов, поэтому регрессии сейчас контролируются ручным и скриптовым прогоном.

## Что было исправлено

1. Добавлен `TaskValidationError`, чтобы отделить ошибки валидации от внутренних ошибок приложения.
2. API `POST /api/tasks` и `PATCH /api/tasks/[id]` теперь возвращают `400` для невалидных данных.
3. Form routes создания и редактирования теперь перехватывают ошибки и возвращают пользователя обратно на форму с параметром `error`.
4. Страницы создания и редактирования читают `error` из query string и показывают сообщение в форме.
5. Redirect'ы после POST-действий переведены на `303`, что безопаснее и предсказуемее после отправки форм.
6. После выявления битого dev-state очищен `.next`, пересобран проект и повторно подтверждён запуск `dev`.
7. Повторно проверены CRUD, поиск, фильтрация, сортировка, overdue и удаление через реальные HTTP-запросы.

## Какие ограничения остались

- статистика на главной странице считается по текущему отфильтрованному набору задач, а не по всей базе;
- валидация отображается только текстовым сообщением без сохранения введённых пользователем значений после неуспешного submit;
- при параллельной работе со старыми dev-процессами и пересборкой может потребоваться очистка `.next`;
- нет автоматических e2e-тестов и test runner'а;
- проект остаётся MVP для лабораторной работы, без авторизации и многопользовательского режима.
